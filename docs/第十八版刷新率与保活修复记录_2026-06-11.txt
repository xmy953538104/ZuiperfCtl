ZuiperfCtl v18 刷新率与保活修复记录 - 2026-06-11

一、修正刷新率学习语义

v17 里把 launcher/default scene 排除在通知栏快速学习之外，这不符合目标行为。
正确语义是:

1. 未配置的场景默认 120Hz。
2. 用户可以给任何可靠前台场景手动学习 60/90/120/144/165Hz，包括桌面。
3. 选择 120Hz 等价于移除该场景的例外规则，回到默认 120Hz。
4. SystemUI 和 ZuiperfCtl 仍视为临时遮罩，不作为学习目标。

已做修改:

- RefreshSceneController.currentLearnPackage 不再排除 launcher。
- zui_perfctld 删除 is_default_scene_package 以及 learn_refresh_default_scene 拒绝分支。
- daemon 仍优先使用 60 秒 TTL 内的 accessibility scene event，再回退到 Settings 里的 top package / last top app。

二、修正保活方式

v17 的 daemon 每 10 秒执行一次 grant_app_permissions + notify_quick_service。
这能自愈无障碍和通知，但长期运行没有必要，也可能造成额外唤醒。

v18 改为:

1. init 阶段仍执行一次授权、无障碍写入和通知启动。
2. main loop 不再周期性重拉前台通知服务。
3. 开机后用一次性 SafeCenter BlackWhite.db 写入解决 ZUI 全部清理保活。
4. 如果开机早期 DB/APK 还没准备好，daemon 只做有限重试，成功或达到上限后停止。

新增路径:

- app/src/main/java/com/zui/perfctl/SafeCenterKeepAlivePatcher.java
- /data/local/tmp/zui_perfctl/safecenter_keepalive_done.flag
- /data/local/tmp/zui_perfctl/safecenter_keepalive_backup.flag
- /data/local/tmp/zui_perfctl/log/safecenter_keepalive.log

四、根目录“最佳路径.txt”评估

评估结论方向正确:

- 最终架构应让 App 只负责 UI/QS/config。
- 前台识别、规则选择、DisplayManager/SurfaceFlinger 操作应进入 system_server 侧服务。
- asouloptd 只负责 asoulopt.conf、线程亲和、vendor 节点等底层执行。
- QS 点击时不应由 App 猜包名，系统服务应使用 lastNonSystemForegroundPackage。

当前可做性:

- 这个方案是长期最佳路径，但不是当前 v18 这种小改能安全完成的内容。
- 本地没有完整 frameworks/base 源码，只有 187 的 framework.jar/services.jar 和既有 smali 反编译产物。
- 在只有 smali 的情况下强行加入 AsoulOptService/Manager API/DisplayModeDirector vote/SurfaceFlinger UID cap，属于高风险 framework patch。
- 当前轮次先保留 accessibility 作为短期场景来源，并解决保活与错误学习问题。

六、注意事项

- 当前 Windows 环境没有 python/bash 命令，因此 py_compile 和 bash -n 仍交给 GitHub Actions 执行。
- v18 daemon 会在首次开机后创建 SafeCenter DB 备份:
  /data/user/0/com.zui.safecenter/databases/BlackWhite.db.bak_zui_perfctl_YYYYMMDD_HHMMSS
- 如果用户 9008 或清空 /data，SafeCenter DB 和 done flag 都会丢失，刷入 v18 后会重新写入。
