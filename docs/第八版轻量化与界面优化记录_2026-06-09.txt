ZuiperfCtl v8 轻量化与界面优化记录
日期：2026-06-09

一、这版目标

1. 研究 ZUI 官方 XML 格式，特别是鸣潮 com.kurogame.mingchao 的 LimitConfig 与 performanceconfig level。
2. 保持系统 App 轻量，不引入 Compose/Hilt/Room 等重依赖，但借鉴 gpt_mobile 的清晰信息层次和横竖屏布局思路。
3. 优化 App 命令回读、通知触发和 daemon 空闲轮询，减少“状态慢半拍”和无意义调度。

二、App 改动

1. 版本升为 0.8.0 / versionCode 8。
2. 顶部状态从固定副标题改为动态状态：
   v0.8.0、当前 peak_refresh_rate、最近命令。
3. 横屏且宽度足够时改为左侧导航 + 右侧内容；竖屏继续顶部页签。
4. 性能页继续左右/上下自适应：
   横屏：左侧配置列表，右侧编辑表单。
   竖屏：上方配置列表，下方编辑表单。
5. 性能页增加短 XML 语义提示：
   三个模式对应 game_policy.xml 的三段 LimitConfig，生成时保留最后一个官方高温保护段。
6. 性能状态框同时显示 XML 挂载状态与生成摘要；保存后但未生成时明确提示“还没有改写工作 XML”。
7. sendCommand 从固定 380ms 回读改成按命令类型等待：
   短命令 720ms。
   apply_performance / restore_zuipp 6500ms。
   export_logs 1800ms。
8. App 不再每个命令后都重启/刷新通知；只有刷新率和状态类命令才主动刷新通知。
9. 通知点击不再乐观地先显示所点刷新率，而是等待系统设置回写后再刷新通知，避免“按钮看起来成功，实际没有规则”的假状态。
10. 通知去掉 customBigContentView 和 DecoratedCustomViewStyle，不再提供无意义的展开大视图。

三、daemon 改动

1. 保存或删除性能配置后清理 summary.txt 和对应 Settings.System 摘要，避免界面显示旧的 XML 生成结果。
2. 无非 120Hz 例外规则时，不再每秒跑前台识别：
   - 每秒只确认 120Hz baseline。
   - 每 5 次 refresh_tick 才更新一次前台记录。
   - 一旦存在例外规则，恢复每秒前台识别，保证应用切换响应。
3. 启动日志改为 zui_perfctld v8 start。
4. 修复通知学习刷新率的假成功：
   - 如果当前前台是通知遮罩或 ZuiperfCtl 本身，会尝试使用最近 15 秒内识别过的真实用户应用。
   - 如果仍找不到可学习应用，60/90/144 不再临时改全局刷新率，而是记录 learn_refresh_no_target。
   - 120Hz 仍允许作为恢复基线操作。

四、XML 文档

新增：
docs/XML格式与鸣潮调参说明_2026-06-09.txt

内容包括：
- game_policy.xml 与 performanceconfig.xml 分工。
- 鸣潮当前官方 LimitConfig 三段模式。
- TempLevel 到温度的映射。
- CPU/GPU Freq 的数值含义。
- 鸣潮官方 level 637-648 的频率解释。
- ThermalConfig 的实际定位：187 游戏名单统一为 200 100 300，是底层 thermal code 组合，不是温度阈值。
- 后续给调参需求时推荐的填写格式。

五、实机只读观察

设备：HA25HSZM / TB321FU / ZUI_16.1.11.187
当前刷机状态仍是 v7，未安装 v8。

观察结果：
- dumpsys cpuinfo 没有把 com.zui.zuiperfctl 或 zui_perfctld 列为持续 CPU 热点；AsoulOpt 为 0%。
- 连续 top 样本中：
  zui_perfctld 多数为 0-2% 瞬时波动。
  com.zui.zuiperfctl 为 0% CPU，RSS 约 159MB。
  AsoulOpt 为 0% CPU，RSS 约 4.5MB。
  com.zui.pp 为 0% CPU，RSS 约 112MB。
- v7 当前性能配置已清空，profiles.prop 为 0 字节。
- 当前 XML 状态：未挂载，使用官方 XML。

判断：
- 当前不是持续高占用。
- 可优化点主要是无规则时的前台轮询、长任务状态回读、通知重复启动和旧 summary 残留。
- v8 已针对这些点处理。
- 通知 60Hz 点击后返回 120 的根因是 v7 记录了 learn_refresh_global：没有学到任何包名，只临时改了全局值，随后守护循环按 120Hz 基线恢复。
